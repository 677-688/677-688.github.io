# 安全性、可靠性与系统性能评测基础知识

## 1. 计算机安全概述

### 1.1 计算机的安全等级

#### 1.1.1 重要的安全评估准则

1、美国国防部和国家标准局推出《可信计算机系统评估准则》（TCSEC）
2、加拿大《可信计算机产品评估准则》（CTCPEC）
3、美国制定的《联邦（最低安全要求）评估准则》（FC）
4、欧洲英、法、德、荷国防部门信息安全 机构联合制定的《信息技术安全评估准则》（ITSEC）
5、美国制定的《信息技术安全评估通用准则》简称CC标准。
#### 1.1.2 TCSEC/TDI将系统划分为4组7个等级

| 组   | 安全级别 | 定义                                     |
| --- | ---- | -------------------------------------- |
| 1   | A1   | 可验证安全性设计。提供B3级保护。                      |
| 2   | B3   | 安全域。TCN必须满足访问监控器的要求，提供系统恢复过程。          |
| 2   | B2   | 结构化安全保护。对系统内的所有主体和客体实施自主访问和强制访问控制      |
| 2   | B1   | 标记安全保护。对系统的数据加以标记。对标记的主体和客体实施强制存取控制。   |
| 3   | C2   | 受访问控制。安全产品的最低 层次，存取控制以用户为单位。           |
| 3   | C1   | 提供了非常初级的自主安全保护，实现用户和数据的分离。数据保护以用户组为单位。 |
| 4   | D    | 最低级别，保护措施很小，没有安全功能                     |
### 1.2 信息安全的基本要素

 - 机密性：确保信息不暴露给未授权的实体和进程。
 - 完整性：只有得到允许的用户才能修改数据，并可以判断数据是否被篡改。
 - 可用性：获得授权的用户可以在需要的时候访问数据。
 - 可控性：可以控制授权范围内的信息流向及行为方式。
 - 可审查性：对出现安全问题可以进行调查和提供依据。
### 1.3 典型的安全威胁

- 授权侵犯：拥有授权的用户，将系统用作其他未授权的目的。
- 拒绝服务：对合理的访问被无条件拒绝。
- 窃听：信息从被监视的通信过程中被泄露出去。
- 信息泄露：信息被泄露或暴露给未授权的实体。
- 截获/修改：数据在传输过程中被改变、修改、删除。
- 假冒：一个实体伪装为另一个实体。
- 否认：参与数据交换后，一方不承认发生过数据交换。
- 非法使用：未授权实体、用户使用了没被授权的资源。
- 人员疏忽：用户由于个人利益、疏忽将数据泄露给未被授权的实体。
- 完整性破坏：对未授权的数据进行创建、修改、破坏。使数据的一致性遭到破坏。
- 媒体清理：信息被 从废弃或打印过的设备中获得。
- 物理入侵：入侵者通过物理控制获得对系统的访问。
- 资源耗尽：某一资源被超负荷使用，造成服务中断。
### 1.4 影响数据安全的因素

#### 1.4.1 内部因素

> 采用加密技术多数据加密；制定数据安全规划；做好数据备份、容错数据保护等；制定数据安全管理规范。
#### 1.4.2 外部因素

> 将数据分成不同的密级，规定外部使用员的权限。比如设置身份认证、密码、设置口令等。设置防火墙防止外部数据入侵破坏数据。
## 2. 加密技术和认证

### 2.1 加密技术

> 加密技术的关键在于加密/解密算法和密钥的管理
> 数据加密的基本过程是对明文的文件或数据按某种加密算法进行处理，使其称为不可读的一段代码，称为密文。
> 密文只能在输入相应的密钥之后才能显示出原本的内容。
#### 2.1.1 对称加密技术（私人密钥加密）

- **特点：文件加密和解密使用相同的密钥**
- **从输入到输出的完整工作流程**
	- **第一步**：输入准备
	- 加密开始前，需要准备三个核心输入：
		1. **明文**：需要被保护的原始数据，可以是任何格式（文本、图片、数据库记录等）。
		2. **密钥**：一串**完全保密**的随机比特序列。它是整个系统安全性的根源。密钥长度直接决定强度（如AES-128使用128位密钥）。
		3. **算法与参数**：包括：
			- **加密算法**：如 AES、ChaCha20。
			- **工作模式**：如 CBC、CTR、GCM，定义了算法如何处理超过单块的数据。
			- **初始化向量（IV）或随机数（Nonce）**：一个**公开但必须唯一**（通常随机）的值，用于确保相同明文在不同次加密中产生不同密文，防止模式分析攻击。
	- **第二步**：加密处理（核心变换）
	- 加密引擎根据所选算法，将明文、密钥和参数进行多轮的、复杂的数学变换。这是算法的核心：
		- **分组密码（如AES）**：将明文分割成固定大小的块（如128位），对每个块应用**代换**和**置换**操作，经过多轮（如10、12、14轮）迭代后输出密文块。模式决定了块之间如何关联。
		- **流密码（如ChaCha20）**：使用密钥和Nonce通过核心函数生成一个伪随机的“密钥流”，然后将其与明文进行简单的**按位异或** 操作，直接得到密文。其安全性取决于密钥流的不可预测性。
	- **第三步**：输出生成
	- 加密过程最终产生两类输出：
		1. **密文**：这是主要输出，外观上是毫无规律的乱码。**没有正确的密钥，密文在计算上是无法被还原为明文的**。
		2. **认证标签**（部分模式）：如果使用 **认证加密模式**（如 **GCM**、**ChaCha20-Poly1305**），加密过程会额外生成一个“消息认证码”作为认证标签。它用于验证密文在传输过程中**未被篡改**，同时确保了完整性。
- **对称加密算法**：

| 算法名称         | 密钥长度           | 分组长度    | 主要特点与优缺点                              | 典型应用场景                                                |
| ------------ | -------------- | ------- | ------------------------------------- | ----------------------------------------------------- |
| **AES**      | 128, 192, 256位 | 128位    | **黄金标准**，安全高效，软硬件实现均优。                | **最广泛**：SSL/TLS、Wi-Fi(WPA2)、文件/磁盘加密(如BitLocker)、政府文档。 |
| **DES**      | 56位            | 64位     | **已过时**，密钥过短，易被暴力破解。                  | **历史遗留系统**，不应在新系统中使用。                                 |
| **3DES**     | 112或168位       | 64位     | DES的临时增强版，三重加密，**慢且逐渐被淘汰**。           | 金融支付系统等旧标准过渡，正被AES取代。                                 |
| **Blowfish** | 32-448位        | 64位     | 密钥灵活，**快**，但**分组短**，不适合加密大文件。         | 旧版密码管理工具、部分嵌入式系统。                                     |
| **Twofish**  | 128, 192, 256位 | 128位    | Blowfish继承者，AES决赛算法，**安全快速**，但未AES普及。 | 关注性能和安全性的替代选择，如某些文件加密软件。                              |
| **ChaCha20** | 256位           | **流密码** | **移动/物联网设备首选**，纯软件实现**极快**，抗侧信道攻击。    | 现代TLS（如QUIC）、移动通信、物联网设备。                              |
| **RC4**      | 40-2048位       | **流密码** | **已破弃用**，存在严重安全漏洞。                    | **绝对禁止**在新系统中使用。                                      |

- **分组密码的工作模式**  
	当使用AES这类分组密码处理长数据时，需要选择模式来定义如何链接各个数据块：
	- **ECB模式**：相同明文块产生相同密文块，**不安全**，不推荐用于实际数据加密。
	- **CBC模式**：每个块与前一密文块异或后再加密，需要**初始向量**，最常用模式之一。
	- **CTR模式**：将计数器加密生成密钥流，再与明文异或，可将分组密码转为流密码，支持并行。
	- **GCM模式**：在CTR基础上增加认证功能，提供**机密性+完整性**，是现代TLS和高速网络的推荐选择。
- **流密码的工作方式**  
	如ChaCha20，直接使用密钥和随机数（Nonce）通过核心算法实时生成伪随机的“密钥流”，然后与明文逐位异或得到密文。其本质是**一次一密**的实用化近似，性能极高。
- **初始化向量与随机数**
	- **IV/Nonce**：对于CBC、GCM等模式，需要一个随机且唯一的初始值，**无需保密**但绝不能重复使用相同密钥时重复，用于保证相同明文加密出不同密文。
	- **填充**：如PKCS#7，用于将数据长度补齐至分组的整数倍。
#### 2.1.2 非对称加密技术（公开密钥加密）

- 与对称加密使用同一把密钥不同，非对称加密的加解密过程是**单向**的;
	- **公钥加密，私钥解密**：用于**加密传输**，任何人可用你的公钥加密信息，只有你可用私钥解密。
	- **私钥签名，公钥验签**：用于**数字签名**，你用私钥对信息生成签名，任何人可用你的公钥验证签名真伪及信息完整性。
#### 总结：对称加密 Vs. 非对称加密

| 特性        | 非对称加密 (如 RSA/ECC)    | 对称加密 (如 AES/ChaCha20) |
| --------- | -------------------- | --------------------- |
| **密钥**    | 密钥对（公钥、私钥）           | 单一共享密钥                |
| **速度**    | **慢**（比对称加密慢1000倍以上） | **极快**                |
| **用途**    | 密钥交换、数字签名、加密小数据      | 加密大量数据                |
| **安全性基础** | 数学难题（分解、离散对数）        | 混淆和扩散                 |

#### 2.1.3 密钥管理

- 使用密码学安全的随机数生成器生成足够长的密钥。
- **密钥必须严格保密**，并定期更换。
- 密钥管理本身通常需要借助非对称加密或硬件安全模块来实现。
### 2.2 认证技术

> 认证技术是确认**用户身份**、**系统实体**或**信息完整性**的技术，是信息安全的基石。它确保“你是你声称的那个人”以及“数据没有被篡改”。主要解决 网络通信过程中通信双方的身份认可。

- 计算机认证主要围绕以下三个核心方面展开：

| 认证类型        | 认证依据                | 典型示例                  | 核心目的                           |
| ----------- | ------------------- | --------------------- | ------------------------------ |
| **身份认证**    | **你是谁** (你的身份)      | 密码、指纹、U盾、数字证书         | 验证用户或系统的**身份**，控制访问权限。         |
| **消息认证**    | **信息是否可信** (完整性+来源) | HMAC、数字签名             | 验证消息在传输中**未被篡改**，且确认**发送者身份**。 |
| **设备/实体认证** | **设备是否合法**          | 802.1X网络接入、SIM卡、TLS握手 | 验证接入网络或服务的**设备或软件实体**是否被授权。    |
#### 2.2.1 身份认证：证明“你是你”

- 这是最常见的认证形式，可分为三类：
 1. **你知道什么**（知识因素）
	- **原理**：基于秘密信息。
	- **示例**：**静态密码**、PIN码、安全问题的答案。
	- **优缺点**：
		- ✅ 成本低，实现简单。
		- ❌ 易被猜解、窃听、钓鱼、撞库攻击。**单一密码认证是弱认证**。
2. **你拥有什么**（持有因素）
	- **原理**：基于物理或逻辑令牌。
	- **示例**：
		- **硬件令牌**：U盾、智能卡、一次性密码器。
		- **软件令牌**：手机APP生成的一次性密码。
		- **密钥文件**：用于SSH登录的私钥文件。
	- **优缺点**：
		- ✅ 安全性显著高于纯密码。
		- ❌ 可能丢失、被盗或需要额外携带设备。
3. **你是什么**（生物特征因素）
	- **原理**：基于独一无二的生理或行为特征。
	- **示例**：**指纹**、**人脸识别**、**虹膜识别**、声纹、笔迹。
	- **优缺点**：
		- ✅ 便捷，难以复制和共享。
		- ❌ 存在误识别率；生物特征一旦泄露无法更改；涉及隐私问题。

**最佳实践：多因素认证**  
将上述两种或多种因素结合使用，安全性呈指数级增长。例如：
- **网银登录**：密码（你知道的）+ 短信验证码（你拥有的手机）。
- **办公室门禁**：工卡（你拥有的）+ 指纹（你是什么）。

#### 2.2.2 消息认证：证明“信息可信”

> 用于确保数据的**完整性**和**来源的真实性**（但不提供机密性）。

1. **消息认证码**
	- **原理**：发送方和接收方**共享一个密钥**。发送方用此密钥和消息通过特定算法（如**HMAC**）计算出一个短小的MAC值，附在消息后。接收方用相同密钥重新计算并比对。
	- **核心**：**共享密钥**。
	- **用途**：API接口鉴权、软件更新包验证、SSL/TLS记录层的完整性保护。
2. **数字签名**
	- **原理**：基于**非对称加密**。发送方用**自己的私钥**对消息的哈希值进行“签名”。接收方用**发送方的公钥**验证签名。
		- **私钥签名，公钥验签**。
	- **核心**：**公私钥对**。解决了MAC需要预先安全分发密钥的问题。
	- **用途**：软件发布签名（验证开发者）、电子邮件签名、区块链交易、SSL/TLS证书。

**消息认证码 vs. 数字签名**

|特性|消息认证码|数字签名|
|---|---|---|
|**密钥类型**|**对称密钥**（共享）|**非对称密钥对**（私钥签名，公钥验证）|
|**证明内容**|消息完整 + 来自**某个共享密钥的持有者**|消息完整 + 来自**私钥的唯一持有者**|
|**密钥分发**|需要安全通道预先分发密钥|公钥可公开分发，无需保密|
|**典型算法**|HMAC-SHA256|RSA签名， ECDSA， EdDSA|

#### 2.2.3 实体认证：证明“设备合法”

> 在系统与系统、设备与网络之间进行认证。

1. **基于证书的认证**
	- **原理**：实体（如Web服务器）持有由可信第三方（**CA**）签发的**数字证书**，内含其公钥和身份信息。在建立连接时出示证书供对方验证。
	- **典型协议**：**SSL/TLS**（HTTPS的基础）。当你访问 `https://` 网站时，浏览器就在验证服务器的证书。
2. **基于预共享密钥的认证**
	- **原理**：双方预先通过安全方式共享一个密钥。
	- **典型场景**：Wi-Fi的WPA2-Personal模式（输入密码）、IPSec VPN的预共享密钥模式。
3. **端口接入控制**
	- **原理**：在网络交换机或无线接入点级别，对接入设备进行认证。
	- **典型协议**：**IEEE 802.1X**。要求设备（如你的电脑）提供身份凭证（如用户名密码或证书）才能接入网络。

#### 2.2.4 核心认证协议与框架
1. **OAuth 2.0**：一个**授权**框架，而非严格的身份认证协议。它允许你授权一个应用（如第三方小程序）在**不获取你密码的情况下**，访问你在另一个服务（如微信）的资源。它常被用作**单点登录**的基础。
2. **OpenID Connect**：基于OAuth 2.0的**身份认证**层。它在授权之余，标准化地返回用户的身份信息，解决了“用户是谁”的问题。
3. **Kerberos**：一种基于**对称加密**的集中式网络认证协议，主要用于企业内部网络，提供“单点登录”体验。
4. **RADIUS**：远程用户拨号认证服务，广泛用于网络设备（路由器、交换机）的集中化用户认证、授权和计费。
    
#### 2.2.5 总结与实践建议

|场景|推荐技术|说明|
|---|---|---|
|**用户登录**|**多因素认证**|至少结合密码+手机令牌/生物特征。|
|**API调用**|**HMAC** 或 **JWT**|确保请求来自合法客户端且未被篡改。|
|**软件/代码发布**|**数字签名**|使用开发者的私钥签名，供用户用公钥验证。|
|**Web安全通信**|**SSL/TLS证书**|强制使用HTTPS，验证服务器身份并加密通信。|
|**内部系统访问**|**单点登录**|使用OIDC、SAML等协议，提升体验与安全管理效率。|

## 3. 计算机可靠性

### 3.1 计算机可靠性概述

> 计算机可靠性是衡量系统**在规定条件下和规定时间内，无故障地完成规定功能**的能力。它是系统设计、评估和维护的核心指标。
#### 可靠性的核心度量指标

1. **可靠度 R(t)**：系统在时间区间 `[0, t]` 内**不发生故障的概率**。它是一个随时间递减的函数。
2. **失效率 λ(t)**：系统工作到时刻 `t` 后，**单位时间内发生失效的概率**。对于电子元件，其“浴盆曲线”通常有早期失效、偶然失效和耗损失效三个阶段。
3. **平均无故障时间 MTBF**：**可修复系统**两次相邻故障间工作时间的平均值。`MTBF = 1/λ`（当λ为常数时）。
4. **平均失效前时间 MTTF**：**不可修复系统**从开始工作到发生故障的平均时间。
### 3.2 计算机可靠性模型

以下是三种最基础的可靠性结构模型，它们是分析更复杂系统的基础。
#### 3.2.1 串联系统

- **定义**：系统中**所有部件都必须正常工作**，整个系统才能正常工作。任一部件失效即导致系统失效。
- **可靠性模型**：`R_total(t) = R₁(t) × R₂(t) × ... × Rₙ(t)`
- **关键结论**：
    - 系统总可靠度**低于**任一单个部件的可靠度。
    - 部件越多，系统可靠度**越低**。
    - 系统失效率 `λ_total ≈ λ₁ + λ₂ + ... + λₙ`。
#### 3.2.2 并联系统（冗余系统）

- **定义**：系统中**只要有一个部件正常工作**，整个系统就能正常工作。所有部件失效系统才失效。
- **可靠性模型**：`R_total(t) = 1 - [ (1 - R₁(t)) × (1 - R₂(t)) × ... × (1 - Rₙ(t)) ]`
- **关键结论**：
    - 系统总可靠度**高于**任一单个部件的可靠度。
    - 并联部件越多，系统可靠度**越高**（但成本也越高）。
    - 这是通过**冗余**提高可靠性的最基本手段。
#### 3.2.3 N模冗余系统

- **定义**：**并联系统的一个特例和增强**。由 `N` 个（通常 `N` 为奇数，如3）**完全相同的模块**并行运行，并由一个**多数表决器**进行输出决策。
- **工作原理**：系统采用“少数服从多数”原则。只要正常工作的模块数 **>** 失效的模块数，表决器就能输出正确结果。
- **可靠性模型**（以最常见的 **三模冗余** 为例）：
    - 假设每个模块的可靠度为 `R`，表决器本身完全可靠。
    - 系统正常工作的情况包括：3个模块全好，或任意2个模块好。
    - `R_TMR = R³ + 3R²(1 - R) = 3R² - 2R³`
- **关键结论**：
    - 当单个模块的可靠度 `R > 0.5` 时，TMR系统的可靠度 `R_TMR > R`，实现了可靠性提升。
    - 当 `R < 0.5` 时，冗余反而会降低可靠性，因此**冗余仅对高可靠度部件有效**。
    - **表决器是关键单点**，其可靠性必须极高，否则将成为系统瓶颈。
#### 模型对比与应用场景

|模型|可靠性逻辑|系统可靠度与部件可靠度的关系|典型应用场景|
|---|---|---|---|
|**串联系统**|**一票否决**|`R_total < min(R₁, R₂...)`|简单链式结构：台式电脑（电源、CPU、内存等缺一不可）、串联电路。|
|**并联系统**|**一票通过**|`R_total > max(R₁, R₂...)`|电源冗余、数据库热备、网络链路聚合、简单备份系统。|
|**N模冗余**|**多数表决**|当`R>0.5`时，`R_total > R`|航天器飞控计算机、高铁信号系统、金融交易核心——任何要求**高容错**的关键实时系统。|
#### 更复杂的模型与现实考量

实际系统往往是**串-并联混合**的复杂结构，需要分解为基本模型进行计算。此外，还需考虑：

1. **冷备份、温备份、热备份**：不同的冗余配置方式，直接影响系统的恢复时间和可靠性。
2. **维修与可修复系统**：引入维修率 `μ`，利用**马尔可夫模型**分析稳态可用度 `A = MTBF / (MTBF + MTTR)`。
3. **共因故障**：冗余部件可能因同一原因（如电源故障、软件bug、环境灾害）同时失效，这会极大削弱冗余效果。工程上需通过**设计多样性**（如硬件异构、软件多版本）来缓解。
4. **可靠性分配与预测**：在系统设计阶段，根据目标可靠度，为各子系统/部件分配合理的可靠性指标。

## 4. 计算机系统的性能评价

### 4.1 性能评价的常用方法

#### 4.1.1 传统性能评价方法（已过时）

这些方法适用于**简单、顺序执行**的早期CPU（如70-90年代的处理器）。

| 方法          | 定义与计算                                                 | 历史意义                | 根本局限与为何过时                                                             |
| ----------- | ----------------------------------------------------- | ------------------- | --------------------------------------------------------------------- |
| **时钟频率**    | CPU内部时钟每秒的周期数，单位Hz（MHz/GHz）。                          | 最直观的参数，曾是市场宣传重点。    | **频率≠性能**。现代CPU通过并行（多核、超标量）和微架构优化（提升IPC）在相同频率下实现数倍性能。单纯比较频率毫无意义。      |
| **执行指令速度**  | **MIPS**：每秒百万条指令。  <br>**MIPS = 指令条数 / (执行时间 × 10⁶)** | 试图衡量CPU“吞吐量”。       | **指令≠工作量**。不同指令复杂度天差地别（一条浮点除法的耗时抵数十条加法）。且不同架构指令集不同，**MIPS完全不可跨架构比较**。 |
| **等效指令速度法** | **吉布森混合法**：给不同指令赋予固定权重，计算“等效指令数”。                     | 试图改进MIPS，考虑指令差异。    | 权重基于特定历史工作负载（如1960年代的科学计算），与**现代应用负载（多媒体、AI）完全脱节**，早已失效。              |
| **数据处理速率**  | **PDR**：基于指令使用频度和操作数位宽计算的加权速率。                        | 曾用于美国政府对计算机的出口控制分级。 | 过于理论化，无法模拟真实程序行为，早已被废弃。                                               |
| **核心程序法**   | 提取实际程序中的**关键循环/核心代码段**进行评测。                           | 向真实评测迈进了一步，比纯理论公式好。 | 代码段太小，无法反映**整个系统**（如缓存、内存、I/O）的性能。易被厂商针对性优化而失真。                       |
**总结**：这些传统方法共同的根本问题是：**试图用一个简单的数字（频率、指令数）来概括复杂的系统性能，忽略了架构差异、程序特性和现代处理器的并行性。**

#### 4.1.2 现代性能评价方法（当前标准）

现代评价完全转向**基于真实或仿真的基准测试**。

| 方法类别              | 核心思想                                    | 典型代表                                                                                              | 特点与用途                                           |
| ----------------- | --------------------------------------- | ------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| **1. 标准基准测试程序套件** | 提供一组**标准化、有代表性的真实程序或核心算法**，在统一规则下运行并计分。 | **SPEC CPU**：评价CPU整数/浮点性能的**行业金标准**。  <br>**其他**：SPECpower（能效）、SPECvirt（虚拟化）、TPC（数据库）、MLPerf（AI）。 | **最权威、最通用**。结果可跨平台比较，被工业界和学术界广泛认可。              |
| **2. 实际应用程序测试**   | 直接使用**目标领域的完整商业/开源软件**进行测试。             | **专业软件**：用Adobe Premiere渲染4K视频计时。  <br>**游戏**：记录游戏帧率（FPS）和帧时间。  <br>**开发工具**：测量项目编译时间。            | **最贴近用户真实体验**。能综合反映CPU、GPU、内存、存储的协同性能。但难以标准化对比。 |
| **3. 性能计数器与剖析**   | 利用CPU内置的硬件计数器，**微观测量**运行时事件。            | **监控指标**：**IPC**、缓存命中率、分支预测失误率、指令吞吐量。                                                             | **用于深度性能分析和优化**，而非给出单一总分。是工程师定位瓶颈的核心工具。         |
| **4. 模拟与仿真**      | 在硬件制造前，用软件模拟整个芯片的设计，预测其性能。              | **工具**：Gem5、SimpleScalar、各种RTL仿真器。                                                                | **用于芯片研发和体系结构研究**，周期极长，但不可或缺。                   |

### 4.2 基准测试程序

| 基准名称                  | 核心目标                                      | 典型应用场景             |
| --------------------- | ----------------------------------------- | ------------------ |
| **SPECvirt / VMmark** | 评估虚拟化平台（Hypervisor）的整体性能，包括CPU、内存、I/O和网络。 | 服务器虚拟化、云计算基础设施选型。  |
| **TPC系列**             | 评估数据库系统的联机事务处理、决策支持等能力。                   | 数据库服务器、企业核心业务系统选型。 |
| **MLPerf**            | 评估AI硬件（GPU/TPU/AI加速卡）在训练和推理任务上的性能。        | AI服务器、边缘AI设备选型。    |
| **3DMark / PCMark**   | 评估PC的图形游戏性能和整体办公、内容创作生产力。                 | 个人电脑、游戏笔记本评测。      |
